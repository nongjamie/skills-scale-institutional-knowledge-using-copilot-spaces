# Role Onboarding Checklist

## Purpose

Streamline onboarding of team members by providing role-specific checklists and orientation resources. This ensures new team members understand their responsibilities, key interactions, and where to find process documentation. Use this guide to accelerate time-to-productivity and ensure consistent onboarding experiences across all roles.

---

## General Onboarding (All Roles)

**Objective**: Ensure all new team members have basic access, understand the project context, and know where to find key resources.

**Timeline**: Typically completed in first 1-2 days

- [ ] **Repository Access**
  - [ ] Access to project repository granted
  - [ ] Local repository cloned and configured
  - [ ] Access to project documentation (Google Drive, Confluence, wiki, etc.)
  - [ ] Read-only access to sensitive systems (as needed)

- [ ] **Communication & Collaboration**
  - [ ] Added to relevant Slack channels and distribution lists
  - [ ] Added to email distribution for project updates
  - [ ] Invited to project kickoff or orientation meeting
  - [ ] Calendar invites sent for recurring meetings (standups, syncs, demos)

- [ ] **Team Introduction**
  - [ ] Introduction to core team members (name, role, how they help)
  - [ ] Introduction to key stakeholders and decision-makers
  - [ ] Assigned a mentor or buddy for questions
  - [ ] Team directory and org chart shared

- [ ] **Framework & Process Overview**
  - [ ] Overview of OctoAcme project management framework
  - [ ] Copy of `docs/octoacme-project-management-overview.md` provided
  - [ ] Project governance and decision-making process explained
  - [ ] Escalation paths and approval workflows clarified

- [ ] **Tools & Access**
  - [ ] Access to project board and tracking tools (GitHub Projects, Jira, etc.)
  - [ ] VPN and security access configured
  - [ ] Required software licenses granted
  - [ ] Development tools or design tools set up (as needed)

- [ ] **Project Context**
  - [ ] Current project status and timeline explained
  - [ ] Project charter / One-pager reviewed
  - [ ] Success metrics and goals understood
  - [ ] Key milestones and delivery dates clarified

- [ ] **Communication Cadence**
  - [ ] Daily standup schedule and format explained
  - [ ] Weekly sync meetings identified
  - [ ] Demo/review schedule communicated
  - [ ] Retrospective timing (if applicable) noted

- [ ] **Completion**
  - [ ] New team member confirms they have all necessary access
  - [ ] New team member confirms understanding of project basics
  - [ ] Mentor/manager schedules follow-up check-in

## Developer Onboarding

**Objective**: Enable developers to contribute code safely and efficiently while following team standards and quality practices.

**Timeline**: Typically completed in first 1-2 weeks (with ongoing learning)

### Development Environment Setup

- [ ] **Local Environment**
  - [ ] Development environment setup guide provided and followed
  - [ ] IDE/editor configured and debugger working
  - [ ] Database or local data set up and verified
  - [ ] Required developer dependencies installed and verified

- [ ] **Repository & Version Control**
  - [ ] Repository cloned and verified
  - [ ] Branch naming conventions documented and understood
  - [ ] Git workflow (rebase vs. merge, commit message format) explained
  - [ ] Access to protected branches configured

- [ ] **Build & Testing**
  - [ ] Build process documented and tested locally
  - [ ] How to run unit tests locally explained
  - [ ] Test coverage expectations clarified
  - [ ] Linting and code formatting tools configured

- [ ] **CI/CD Pipeline**
  - [ ] CI/CD pipeline overview provided
  - [ ] How to view build status and logs explained
  - [ ] What triggers automated tests and deployments clarified
  - [ ] Troubleshooting common CI failures documented

### Code Quality & Review

- [ ] **Code Standards**
  - [ ] Code style guide and conventions documented
  - [ ] Linting rules configured and understood
  - [ ] Naming conventions for variables, functions, classes clarified
  - [ ] Documentation and comments standards explained

- [ ] **Code Review Process**
  - [ ] PR submission guidelines documented
  - [ ] Code review expectations (response time, approval process) clarified
  - [ ] How to request reviews and handle feedback explained
  - [ ] Merge policy (1 approval, 2 approvals, etc.) confirmed

- [ ] **Definition of Done**
  - [ ] Definition of Done checklist documented and reviewed
  - [ ] Code review checklist provided
  - [ ] Testing requirements (unit, integration) clarified
  - [ ] Documentation requirements explained

### Technical Knowledge

- [ ] **Architecture & Tech Stack**
  - [ ] System architecture overview provided
  - [ ] Key components and their responsibilities explained
  - [ ] Technology stack and why each tool was chosen discussed
  - [ ] Data flow and integration points clarified

- [ ] **Design Patterns & Best Practices**
  - [ ] Common design patterns used in codebase explained
  - [ ] Anti-patterns to avoid documented
  - [ ] Performance considerations and optimization guidelines discussed
  - [ ] Security best practices specific to this project covered

- [ ] **Key Systems & Dependencies**
  - [ ] External APIs and how they're used explained
  - [ ] Database schema overview provided (if applicable)
  - [ ] Message queues, caching layers, or other infrastructure explained
  - [ ] Monitoring and logging practices described

### Practical Experience

- [ ] **First Task**
  - [ ] First task/issue assigned (small, well-documented issue preferred)
  - [ ] Task context and acceptance criteria reviewed
  - [ ] Pair programming session scheduled (optional but recommended)
  - [ ] Expected effort and timeline discussed

- [ ] **Pairing & Mentoring**
  - [ ] Paired with existing developer on first feature or fix
  - [ ] Code review feedback received and discussed
  - [ ] Architecture review or design discussion held
  - [ ] Mentor available for questions during first week

### Security & Operations

- [ ] **Security & Access Control**
  - [ ] Security policies and access controls explained
  - [ ] Secrets management and credential handling documented
  - [ ] Code scanning and security testing procedures explained
  - [ ] Incident response process for security issues clarified

- [ ] **Deployment & Operations**
  - [ ] Deployment process and approval workflow explained
  - [ ] Hotfix procedures documented
  - [ ] Rollback procedures explained
  - [ ] On-call responsibilities and procedures (if applicable) covered

**Key Resources to Share:**

- `docs/octoacme-execution-and-tracking.md` — PR workflow, quality standards, blockers
- `docs/octoacme-release-and-deployment.md` — release process, deployment procedures
- Code repository README with setup instructions
- Architecture decision records (ADRs) or design documentation
- Style guide and code standards documentation
- Monitoring and logging dashboards

## Tips for Using This Onboarding Checklist

### For Managers/Mentors

1. **Customize for Your Context**: Modify the checklists based on your specific project, team, and organization needs
2. **Assign a Buddy/Mentor**: Designate someone to help the new team member navigate the first days
3. **Break Into Phases**: Spread onboarding across the first 1-3 weeks rather than trying to do everything on day 1
4. **Check In Regularly**: Use 1-week, 2-week, and monthly check-ins to ensure the person is ramped up and engaged
5. **Collect Feedback**: Ask the new team member to provide feedback on the onboarding process to help improve it for future hires

### For New Team Members

1. **Ask Questions**: Don't hesitate to ask for clarification or help—that's what your mentor/buddy is for
2. **Take Notes**: Keep notes on what you learn for future reference
3. **Schedule Time**: Block time on your calendar to complete onboarding tasks
4. **Introduce Yourself**: Don't wait to be introduced—reach out to team members and stakeholders
5. **Provide Feedback**: Share feedback on your onboarding experience to help improve the process

### General Tips

- **Make It Accessible**: Keep onboarding checklists in a shared location (GitHub, Confluence, Google Docs)
- **Update Regularly**: Review and update onboarding checklists quarterly based on feedback
- **Link to Resources**: Ensure all documentation references are current and accessible
- **Consider Async**: Some items can be completed asynchronously (watching videos, reading docs) before synchronous meetings
- **Celebrate Milestones**: Acknowledge when someone completes onboarding and is ready to contribute!
